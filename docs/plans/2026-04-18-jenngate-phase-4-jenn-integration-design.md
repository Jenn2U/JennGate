# Phase 4: JennGate Integration with Jenn Production — Design

**Date:** 2026-04-18  
**Status:** Design Approved  
**Decision:** Full cutover to JennGate, clean break from Jenn local gate services

---

## Context

Phase 3b (JennGate standalone service) is complete with dual-protocol GUI access (SSH terminal + VNC + X11), database schema, services, REST API, daemon implementations, and 14-item pre-release validation checklist.

Phase 4 integrates JennGate into Jenn Production by:
1. Migrating all gate functionality from Jenn's local services to JennGate
2. Updating three clients (JennSentry CLI, Jenn UI, iJENN2u mobile) to call JennGate APIs
3. Establishing mTLS authentication between Jenn and JennGate
4. Implementing staged rollout with rollback capability
5. Ensuring zero downtime cutover (maintenance window, no active sessions)

---

## Architecture: Clean Separation of Concerns

```
┌─────────────────────────────────────┐
│       Jenn Production (UI)          │
│  - Admin panels (device approval)   │
│  - Policy management & sync         │
│  - Session viewing                  │
└──────┬──────────────────────────────┘
       │ mTLS (service cert) requests
       ↓
┌─────────────────────────────────────┐
│      JennGate Service (Go)          │
│  - Device registry & approval       │
│  - SSH cert issuance                │
│  - Session lifecycle                │
│  - Recording storage (NAS/disk)     │
└──────┬──────────────────────────────┘
       │ gRPC/WebSocket
       ↓
┌─────────────────────────────────────┐
│     Clients (Direct to JennGate)    │
│  - JennSentry CLI (SSH+VNC)         │
│  - Jenn UI (SSH+VNC+X11)            │
│  - iJENN2u mobile (SSH only)        │
└─────────────────────────────────────┘
```

**Key principles:**
- JennGate is authoritative for all remote access operations
- Jenn provides management/governance (device approval, policy ownership)
- Clients connect directly to JennGate (no Jenn proxy)
- Policies flow Jenn → JennGate (Jenn is source of truth)

## 1. Service Certificate & PKI Management

**mTLS between Jenn ↔ JennGate:**

- **Certificate Authority:** JennAuth's existing CA (reuse for consistency across ecosystem)
- **Jenn Service Certificate:**
  - Long-lived (1 year TTL)
  - Subject: CN=`jenn.internal`
  - Use case: Jenn authenticates to JennGate for API calls
  - Generated by: JennAuth CA (request via JennAuth admin API)
  - Storage: GitHub Org Secrets encrypted, deployed via CI/CD
  
- **JennGate Server Certificate:**
  - Subject: CN=`jenngate.internal`
  - Signed by: JennAuth CA
  - Storage: Docker secrets / Kubernetes ConfigMap

- **Certificate Rotation:**
  - Automated 30 days before expiry
  - Generated by JennAuth CA
  - Deployed via GitHub Actions to both services
  - No manual intervention required

**Prerequisite before Phase 4 cutover:**
- Generate Jenn service certificate via JennAuth admin panel
- Configure Jenn to load certificate from secrets
- Configure JennGate to trust JennAuth root CA
- Test mTLS handshake in staging environment
- Document certificate lifecycle in ops runbook

## 2. Recording Storage Architecture

**Session recordings captured on daemons via `script(1)` typescript:**

- **Capture location:** Target daemon (JennEdge/JennDock)
- **Storage location:** NAS mount `/mnt/jenngate/recordings` (or configurable per deployment)
- **Lifecycle:**
  - Daemon captures typescript while session is active
  - On session end, daemon uploads to JennGate storage API
  - JennGate stores in `/var/lib/jenngate/recordings` (or mounted NAS for HA)
  - Compression: gzip (default)
  - Retention: 30 days (configurable per policy)
  - Deletion: On device decommission, explicit admin deletion, or retention expiry

- **Access:** Read-only via JennGate API: `GET /api/v1/gate/recordings/:recording_id`
- **Audit:** All recording access logged (who viewed, when, which recording)

**Pre-cutover actions:**
- Delete all old Jenn gate recordings (as decided — clean break)
- Provision NAS storage quota for JennGate (estimate: 100 GB for 30-day retention, adjust based on device count)
- Configure recording upload destination in all daemons

## 3. Policy Sync (Jenn → JennGate)

**Jenn is source of truth for access policies:**

- **Sync mechanism:** Jenn calls JennGate gRPC API `SyncAccessPolicies(policies, timestamp)`
- **Frequency:**
  - Real-time: Immediate push on policy creation/modification/deletion
  - Reconciliation: Hourly full sync (ensures consistency if any message lost)
  
- **Conflict resolution:** JennGate always accepts Jenn's version (last-write-wins, Jenn wins)
- **Failure handling:**
  - First attempt: Immediate retry (exponential backoff 1s, 2s, 4s, 8s)
  - Persistent failure: Alert Jenn ops, manual intervention required
  - Grace period: 30 seconds (if policy push fails within 30s, rollback policy change in Jenn)

- **Backward compatibility:** Old policies in Jenn (pre-cutover) included in initial sync migration
- **Audit:** All policy sync events logged (timestamp, policy count, success/failure, operator)

## 4. Daemon Compatibility & Pre-Cutover Updates

**All daemons must support JennGate RPC before cutover:**

- **Requirement:** Daemons must implement:
  - `RegisterDaemon()` — report to JennGate on startup
  - `NotifyGUISessionStart/End()` — inform JennGate when GUI server starts/stops
  - `FetchPolicies()` — download access policies for local evaluation
  - `ReportSessionStart/End()` — report session lifecycle events

- **Phase 3b daemon binaries** are ready; version 3.1.0 or later required

- **Pre-cutover deployment sequence:**
  1. **Week 1 — Canary update:** Select 2-3 JennEdge/JennDock nodes
     - Deploy new daemon binaries
     - Verify successful registration with JennGate
     - Monitor for 48 hours
  
  2. **Week 1 (continued) — Staged rollout:** Deploy to remaining devices
     - 25% → 50% → 75% → 100%
     - Monitor health metrics between each stage
     - Verify policy sync, session creation working
  
  3. **Week 2 — Validation:** All daemons on new version, ready for Jenn cutover

## 5. Staging Integration Testing (3-5 days)

**Before production cutover, run comprehensive integration tests in staging:**

**Setup:**
- Staging Jenn + Staging JennGate running simultaneously
- 5-10 test daemons registered with Staging JennGate
- Jenn connected to Staging JennGate via mTLS

**Test scenarios (all must pass):**

1. **Device Registration:** Daemon starts → reports to JennGate → appears as PENDING_APPROVAL
2. **Device Approval:** Jenn admin approves device → JennGate updates state → policies synced
3. **Policy Sync:** Create policy in Jenn → push to JennGate → verify daemon sees updated policy
4. **Certificate Issuance:** User requests cert → Jenn calls JennGate → cert issued with correct TTL
5. **Session Lifecycle:** Create session → connect via WebSocket → run commands → disconnect → session recorded
6. **VNC Access:** Request enable_gui=true for VNC → daemon starts VNC server → user connects via port forwarding
7. **X11 Access:** Request enable_gui=true for X11 on JennEdge → Xvfb starts → user connects with -X forwarding
8. **WebSocket Terminal:** JennSentry CLI connects directly to JennGate WS → run remote commands
9. **Error Handling:** 
   - Policy denies access → cert request rejected
   - Device offline → session fails gracefully
   - Certificate expired → reconnect fails with clear error
10. **Audit Logging:** Verify all state changes logged (device approval, policy sync, session lifecycle, errors)

**Load testing:**
- 10 concurrent sessions
- Policy sync under 5 seconds
- Certificate issuance under 100ms
- WebSocket latency < 50ms

**Success criteria:** All 10 scenarios pass, load test succeeds, no data loss, audit logs complete

## 6. Rollback Strategy

**If production cutover encounters critical issues:**

**Immediate actions (< 15 min):**
1. Stop routing new traffic to JennGate (restore Caddy config to old Jenn gate routes)
2. Notify all users (maintenance window in progress, reverting to previous system)
3. Preserve JennGate state (don't delete DB, keep logs for post-mortem)

**Recovery (15-60 min):**
1. Re-enable old Jenn gate services (gate_routes.py, gate_*_service.py)
2. Restore daemon RPC endpoints to old Jenn gRPC endpoints
3. Resume operations with old system
4. Notify users (system stable, cutover being retried)

**Post-mortem (within 24 hours):**
1. Investigate JennGate logs (find root cause)
2. Analyze staging test results (why tests didn't catch issue)
3. Fix issues in JennGate code
4. Schedule retry cutover (1-2 weeks, after additional testing)
5. Document lessons learned

**Rollback eligibility:** Allowed anytime up to 24h post-cutover. After 24h, rolling forward is preferred (fix issues in JennGate rather than reverting).

## 8. Client-Specific Implementation

### **JennSentry CLI (Phase 1 — Week 3, Day 3)**

**Update `jennctl gate` commands:**
- `jennctl gate access <device>` — request access from JennGate
- `jennctl gate connect <device>` — establish SSH or VNC session
- `jennctl gate sessions` — list active sessions
- `jennctl gate recordings <device>` — view recordings

**Protocol support:** SSH terminal + VNC (no X11 for CLI)

**Connection flow:**
1. User runs `jennctl gate access mydevice`
2. JennSentry calls `POST /api/v1/gate/cert/issue` (mTLS to JennGate)
3. JennGate validates user permissions, issues ephemeral SSH cert
4. JennSentry gets WebSocket URL: `wss://jenngate.internal:8081/ws/gate/:session_id`
5. JennSentry connects directly to JennGate WebSocket
6. User has interactive SSH session (xterm.js in CLI)
7. For VNC: Port forwarding via SSH ProxyJump to JennGate, connect VNC client to localhost:5900

**WebSocket endpoint:** `wss://jenngate.internal:8081/ws/gate/:session_id`

### **Jenn Production UI (Phase 2 — Week 3, Day 4)**

**Admin panels (unchanged except routing):**
- Device management: approval, decommission (call JennGate `/admin/gate/` endpoints)
- Policy management: create, update, delete (call JennGate `/api/v1/gate/policies` endpoints)
- Session viewing: active sessions, recordings (call JennGate `/api/v1/gate/sessions`)

**User panels (new, calls JennGate):**
- "Request Access" button → `POST /api/v1/gate/cert/issue` → get certificate PEM
- "Open Terminal" → WebSocket to `wss://jenngate.internal:8081/ws/gate/:session_id`
- "Open VNC" (if enabled) → SSH port forwarding + VNC client
- "View Recordings" → download from JennGate API

**Protocol support:** SSH terminal + VNC + X11 (selective based on device type and user permissions)

**WebSocket endpoint:** Same as JennSentry

### **iJENN2u Mobile (Phase 3 — Week 4, deferred)**

**Update mobile app:**
- Call JennGate REST APIs for certificate issuance
- WebSocket connection to JennGate for SSH terminal
- Defer until 24h of stability post-cutover (lower risk, last client)

**Protocol support:** SSH terminal only (mobile UI constraints, no VNC/X11)

**WebSocket endpoint:** Same as others

## 9. API Contract Changes & Backward Compatibility

**Breaking changes from old Jenn gate API:**

| Old Jenn API | JennGate API | Change | Migration |
|--------------|--------------|--------|-----------|
| POST `/api/v1/gate/cert/issue` with `public_key` param | POST `/api/v1/gate/cert/issue` with `device_id` param | Simplified; cert issued for authenticated user | Update all clients to use new format |
| Response: `certificate`, `serial`, `expires_at` | Response: same + `gui_available`, `vnc_port`, `x11_display` | New fields for GUI support | Old clients ignore new fields (backward compat) |
| WS `/ws/gate/:session_id` | WS `/ws/gate/:session_id` | Same endpoint path | No change needed |

**Migration plan:**
1. Update JennSentry to use new API (Phase 1)
2. Update Jenn UI to use new API (Phase 2)
3. Update iJENN2u to use new API (Phase 3)
4. After all clients updated + 7 day grace period, remove old Jenn gate code
5. Update `CROSS_PROJECT_CONTRACT.md` Section 7 with JennGate API surfaces

## 13. Risk Mitigation

| Risk | Impact | Likelihood | Mitigation |
|------|--------|-----------|-----------|
| mTLS cert generation fails | Complete outage | Low | Pre-generate + test in staging |
| Policy sync loses data | User access denied | Low | Jenn is source of truth, hourly reconciliation |
| WebSocket drops connections | Session lost | Medium | Auto-reconnect logic in clients |
| Daemon incompatibility | Device offline | Low | Pre-cutover daemon update phase |
| Recording storage full | Recordings lost | Low | Monitor disk 80% alert, auto-cleanup old |
| Rollback needed mid-cutover | Extended downtime | Medium | Documented procedure, old services available 1 week |
| Performance degradation | Slow access | Medium | Staging load tests, pre-cutover optimization |
| Certificate expiry during migration | Auth failure | Very Low | 1 year TTL, automated rotation 30 days before |
